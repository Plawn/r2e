# r2e-grpc

gRPC server support for R2E ŌĆö Tonic-based, with the same DX as HTTP controllers.

## Overview

Host gRPC services alongside HTTP controllers with full access to R2E's DI, guards, interceptors, and identity extraction. Supports separate-port and multiplexed (same-port) transport modes.

## Usage

Via the facade crate:

```toml
[dependencies]
r2e = { version = "0.1", features = ["grpc"] }

[build-dependencies]
r2e-grpc-build = "0.1"
```

### Compiling protos

Drop your `.proto` files under `proto/` and add a one-line build script. The
`r2e-grpc-build` helper compiles every `proto/**/*.proto` (via
`tonic-prost-build`), then emits an aggregated per-package module plus a
combined `FILE_DESCRIPTOR_SET` into `OUT_DIR`:

```rust
// build.rs
fn main() -> Result<(), Box<dyn std::error::Error>> {
    r2e_grpc_build::compile()
}
```

The proto directory is registered with `cargo:rerun-if-changed`, so adding,
editing, or removing a `.proto` is picked up automatically ŌĆö no `build.rs`
change. An empty/missing `proto/` is not an error (a stub module is emitted).
`ProtoCompiler::new().proto_dir(...).configure(...)` is the escape hatch for a
custom proto directory or `tonic-prost-build` tweaks.

### Defining a service

Include the generated modules with `include_protos!()`, then attach your impl
with `#[grpc_routes]`:

```rust
pub mod proto {
    r2e::r2e_grpc::include_protos!();
}

use proto::greeter::{HelloReply, HelloRequest};

#[grpc_routes(proto::greeter::greeter_server::Greeter,
              descriptor = proto::FILE_DESCRIPTOR_SET)]
impl GreeterService {
    // tonic service methods, with #[inject]/#[config] on the struct
}
```

The `descriptor = ...` argument is optional ŌĆö pass it to enable gRPC server
reflection (`GrpcServer::with_reflection()`, feature `grpc-reflection`).

### Separate port

```rust
use r2e_grpc::GrpcServer;

AppBuilder::new()
    .plugin(GrpcServer::on_port("0.0.0.0:50051"))
    .build_state()
    .await
    .register_grpc_service::<UserGrpcService>()
    .serve("0.0.0.0:3000")   // HTTP on 3000, gRPC on 50051
    .await;
```

### Multiplexed (same port)

```rust
AppBuilder::new()
    .plugin(GrpcServer::multiplexed())
    .build_state()
    .await
    .register_grpc_service::<UserGrpcService>()
    .serve("0.0.0.0:3000")   // both HTTP and gRPC on 3000
    .await;
```

## Key types

| Type | Description |
|------|-------------|
| `GrpcServer` | `PreStatePlugin` ŌĆö configures transport mode (+ `with_reflection`) |
| `#[grpc_routes]` | Attribute macro on the tonic-trait impl; generates the `GrpcService` impl |
| `GrpcService` | Trait wiring a service into R2E (implemented via `#[grpc_routes]`) |
| `include_protos!()` | Includes the modules generated by `r2e-grpc-build` (+ `FILE_DESCRIPTOR_SET`) |
| `AppBuilderGrpcExt` | Extension trait adding `.register_grpc_service::<T>()` |
| `GrpcGuard` / `GrpcGuardContext` | Authorization guards for gRPC methods |
| `GrpcRolesGuard` | Role-based access control for gRPC |
| `GrpcIdentityExtractor` | Extract JWT identity from gRPC metadata |

## Features

- One-line proto compilation via `r2e-grpc-build` + `include_protos!()`
- Compile-time dependency check (missing bean = compile error at `register_grpc_service`)
- `#[inject]`, `#[config]` on gRPC service structs
- Guards and interceptors via `#[guard]` / `#[intercept]`
- JWT identity extraction from `authorization` metadata
- gRPC server reflection (feature `grpc-reflection`)
- Graceful shutdown with drain support (awaited alongside the HTTP drain)
