# Phase 3 — Named/Qualified Beans & Lazy Initialization

**Depends on:** Phase 1 (Option<T>), Phase 2 (profiles/alternatives)

## Goal

Support multiple beans of the same underlying type (qualified by name) and deferred construction for expensive beans.

---

## Feature A: Named / Qualified Beans

### Problem

Currently, `BeanContext` is keyed by `TypeId`. You cannot have two `SqlitePool` instances (e.g., primary and analytics DBs). Workarounds today:
- Newtype wrappers (`struct PrimaryPool(SqlitePool)`) — works but verbose
- Separate types — forced divergence

### User-facing API

```rust
// Named producers
#[producer(name = "primary")]
async fn primary_pool(#[config("db.primary.url")] url: String) -> SqlitePool {
    SqlitePool::connect(&url).await.unwrap()
}

#[producer(name = "analytics")]
async fn analytics_pool(#[config("db.analytics.url")] url: String) -> SqlitePool {
    SqlitePool::connect(&url).await.unwrap()
}

// Named injection in bean constructors
#[bean]
impl UserRepository {
    fn new(#[inject(name = "primary")] pool: SqlitePool) -> Self {
        Self { pool }
    }
}

#[bean]
impl AnalyticsService {
    fn new(#[inject(name = "analytics")] pool: SqlitePool) -> Self {
        Self { pool }
    }
}

// Builder
AppBuilder::new()
    .with_producer::<PrimaryPool>()     // named "primary"
    .with_producer::<AnalyticsPool>()   // named "analytics"
    .with_bean::<UserRepository>()
    .with_bean::<AnalyticsService>()
    .build_state::<AppState, _, _>().await
```

### Design Options

#### Option 1: Newtype wrapper (generated by macro) — RECOMMENDED

The `#[producer(name = "primary")]` macro generates a newtype:

```rust
// Generated:
#[doc(hidden)]
pub struct __Named_primary_SqlitePool(pub SqlitePool);

impl Clone for __Named_primary_SqlitePool { ... }
impl Deref for __Named_primary_SqlitePool { type Target = SqlitePool; ... }
```

And the `#[inject(name = "primary")]` on a `SqlitePool` param generates:

```rust
let pool: SqlitePool = ctx.get::<__Named_primary_SqlitePool>().0;
```

**Pros:**
- No change to `BeanContext` (still `HashMap<TypeId, ...>`)
- Compile-time safety preserved — each named bean is a distinct type
- `BuildableFrom` / `Contains` work naturally
- Zero runtime cost (newtype is erased)

**Cons:**
- Generated type names are ugly (but `#[doc(hidden)]`)
- Cross-crate naming: the newtype must be accessible where it's consumed
- Name collisions: need a naming convention (module-scoped)

**Cross-crate solution:** The newtype is generated in the module where `#[producer(name = "...")]` is defined. Consumers import it. The macro could generate a path like `<module>::named::<Name><Type>`.

Alternative: use a generic wrapper `Named<const N: &'static str, T>` (requires `adt_const_generics` — unstable as of Rust 1.85). Not viable today.

**Practical approach:** Generate the newtype in a predictable submodule:

```rust
// Generated by #[producer(name = "primary")] on SqlitePool:
pub mod __r2e_named {
    #[derive(Clone)]
    pub struct Primary<T>(pub T);
    // The actual registered type is Primary<SqlitePool>
}
```

Or simpler — the user defines the newtype manually (zero magic):

```rust
#[derive(Clone)]
pub struct PrimaryPool(pub SqlitePool);

#[derive(Clone)]
pub struct AnalyticsPool(pub SqlitePool);

#[producer]
async fn primary(#[config("db.primary.url")] url: String) -> PrimaryPool {
    PrimaryPool(SqlitePool::connect(&url).await.unwrap())
}
```

**Recommendation:** Start with explicit newtypes (no macro magic). Then optionally add `#[producer(name = "...")]` sugar that generates them. This keeps the system simple and is already possible today — the "Phase 3" part is the ergonomic sugar.

#### Option 2: Composite key in BeanContext

Change `BeanContext` from `HashMap<TypeId, ...>` to `HashMap<(TypeId, Option<&'static str>), ...>`.

```rust
impl BeanContext {
    pub fn get_named<T: Clone + 'static>(&self, name: &str) -> T { ... }
    pub fn try_get_named<T: Clone + 'static>(&self, name: &str) -> Option<T> { ... }
}
```

**Pros:**
- No newtypes needed
- Clean API

**Cons:**
- Breaks compile-time safety — names are strings, not types
- `BuildableFrom` can't track named beans (no type to match)
- `Contains` trait only works with types
- Requires runtime name validation

**Verdict:** Breaks the compile-time guarantee model. NOT recommended unless we accept that named beans are runtime-only.

#### Option 3: Hybrid — type-level names via const generics (future Rust)

```rust
pub struct Named<const NAME: &'static str, T>(pub T);
// Named<"primary", SqlitePool> is a distinct type from Named<"analytics", SqlitePool>
```

Requires `adt_const_generics` stabilization. Track rust-lang/rust#95174.

### Implementation (Option 1 — Newtype sugar)

#### 1. Macro changes for `#[producer(name = "...")]`

In `producer_attr.rs`, when `name` attribute is present:
- Generate a newtype: `pub struct {PascalName}{ReturnType}(pub {ReturnType})`
- The producer's `Output` type becomes the newtype
- The factory wraps the result in the newtype

#### 2. Macro changes for `#[inject(name = "...")]`

In `bean_attr.rs` and `bean_derive.rs`:
- When `#[inject(name = "primary")]` is on a `SqlitePool` param/field:
- Generate: `ctx.get::<PrimaryPool>().0` (or `.into()`)
- Add `PrimaryPool` to deps instead of `SqlitePool`

#### 3. Name resolution

The newtype must be importable by consumers. Convention:
- Generated in same module as the producer
- Exported with `pub` visibility
- Consumers import: `use crate::producers::PrimaryPool;`

## Feature B: Lazy Beans

### Problem

Some beans are expensive to construct (e.g., ML model loading, large cache warm-up) but may not be used on every request path. Currently, all beans are constructed eagerly during `build_state()`.

### User-facing API

```rust
#[bean(lazy)]
impl ExpensiveService {
    async fn new(pool: SqlitePool) -> Self {
        // Heavy initialization...
        Self { model: load_model().await }
    }
}

// Consumer — transparent, same as non-lazy:
#[bean]
impl MyController {
    fn new(service: Lazy<ExpensiveService>) -> Self {
        Self { service }
    }

    async fn handle(&self) {
        // First access triggers construction
        let svc = self.service.get().await;
        svc.do_work();
    }
}
```

### Design

#### `Lazy<T>` wrapper type

```rust
// r2e-core/src/lazy.rs
pub struct Lazy<T: Clone + Send + Sync + 'static> {
    inner: Arc<OnceCell<T>>,
    factory: Arc<dyn Fn() -> Pin<Box<dyn Future<Output = T> + Send>> + Send + Sync>,
}

impl<T: Clone + Send + Sync + 'static> Lazy<T> {
    pub async fn get(&self) -> &T {
        self.inner.get_or_init(|| (self.factory)()).await
    }
}

impl<T: Clone + Send + Sync + 'static> Clone for Lazy<T> { ... }
```

#### Registration

`#[bean(lazy)]` changes the registered type from `T` to `Lazy<T>`:
- The factory captures the `BeanContext` snapshot at resolve time
- Construction is deferred to first `.get().await`

```rust
// In BeanRegistry, for lazy beans:
self.beans.push(BeanRegistration {
    type_id: TypeId::of::<Lazy<T>>(),  // registered as Lazy<T>, not T
    // ...
    factory: Box::new(|ctx| {
        Box::pin(async move {
            let lazy = Lazy::new(move || {
                Box::pin(async move { T::build(&ctx) })
            });
            (ctx, Box::new(lazy))
        })
    }),
});
```

#### Compile-time safety

- `Lazy<T>` is a distinct type in `P` — consumers must declare `Lazy<T>`, not `T`
- `BuildableFrom` bound is on `Lazy<T>`
- This is explicit: you know at the type level that construction is deferred

#### `PostConstruct` interaction

Lazy beans' `#[post_construct]` hooks should run on first access, not during `build_state()`. The `Lazy` wrapper captures the hook and runs it after construction.

### Implementation

#### 1. Add `Lazy<T>` type — `r2e-core/src/lazy.rs`

New module with the `Lazy` struct, `Clone`, `Send`, `Sync` impls.

Use `tokio::sync::OnceCell` for async initialization.

#### 2. Macro changes

`#[bean(lazy)]` modifies codegen:
- Registered type: `Lazy<Self>` instead of `Self`
- Factory captures the BeanContext and defers construction
- `Deps` still declares dependencies (they must be available at resolve time for the closure)

#### 3. Builder method

```rust
pub fn with_lazy_bean<B: AsyncBean>(mut self) -> AppBuilder<NoState, TCons<Lazy<B>, P>, ...> {
    self.shared.bean_registry.register_lazy::<B>();
    self.with_updated_types()
}
```

## Comparison: Phase 3 Features

| Feature | Compile-time safe | Runtime overhead | Complexity |
|---------|-------------------|------------------|------------|
| Named beans (newtype) | Yes | Zero (newtype erased) | Low (macro sugar) |
| Named beans (composite key) | No | HashMap lookup | Medium |
| Lazy beans | Yes (`Lazy<T>` in type) | Arc + OnceCell | Medium |

## Test Plan

### Named Beans
1. Two producers with different names for same type → distinct beans
2. `#[inject(name = "...")]` resolves the correct named bean
3. Missing named bean → compile error (type not in P)
4. Cross-module named bean usage

### Lazy Beans
5. Lazy bean not constructed during `build_state()`
6. First `.get().await` triggers construction
7. Second `.get().await` returns cached value
8. Lazy bean's deps must be present at resolve time
9. PostConstruct on lazy bean runs on first access

## Open Questions

1. **Named beans naming convention:** Should the generated newtype be `{Name}{Type}` or `Named{Name}` or user-chosen?
2. **Lazy + Option:** Should `Option<Lazy<T>>` be supported? (conditional + lazy)
3. **Lazy + PostConstruct ordering:** If bean A depends on Lazy<B> and has a post_construct that calls B.get(), should that be allowed? (It would trigger B's construction inside A's hook.)
4. **Const generic names:** Monitor `adt_const_generics` stabilization — if it lands, `Named<"primary", T>` becomes the cleaner solution.
